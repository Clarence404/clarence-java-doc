# 项目亮点与难点

> 面试中"说说你项目的亮点/难点"是必考题。本文整理常见亮点方向与表达框架，帮助从技术深度和业务价值两个维度提炼项目经验。

---

## 一、亮点提炼框架

好的亮点描述应包含四要素：**背景（问题）→ 方案 → 实现细节 → 量化结果**

```
背景：系统遇到了什么问题，规模是多大（QPS/数据量/用户量）
方案：选择了什么技术手段解决，为什么选这个而不是其他
细节：具体怎么做的，有哪些技术难点
结果：优化前后对比数据（响应时间/吞吐量/成本）
```

**示例结构：**
> "我们的订单系统在大促期间峰值 QPS 达到 2w，原来直接写 MySQL 导致 P99 延迟 800ms。我们引入了 RocketMQ 做异步削峰，订单请求先写 Redis 预占库存（10ms 内返回），异步落库。同时对热点商品做了本地缓存（Caffeine），缓存命中率达到 95%，接口 P99 降到 30ms 以内。"

## 二、常见亮点方向

### 2.1 性能优化类

**缓存优化：**
- 引入 Redis / 本地缓存 Caffeine，接口响应从 200ms 降到 10ms
- 两级缓存架构（Caffeine + Redis），缓解 Redis 热点压力
- 解决缓存穿透（布隆过滤器）/ 击穿（互斥锁）/ 雪崩（随机 TTL + 熔断）

**数据库优化：**
- 慢 SQL 治理：通过 EXPLAIN 分析，添加联合索引，查询从 5s 降至 20ms
- 读写分离，主库写压力降低 70%
- 历史数据归档，主表数据量从 10亿 降至 1亿，查询速度提升 5 倍

**异步化：**
- 同步调用改 MQ 异步，接口响应时间从 1s 降至 100ms
- 耗时操作（发短信/发邮件/更新统计）异步解耦，核心链路提速

### 2.2 高可用类

**熔断降级：**
- 接入 Sentinel，对三方服务设置熔断阈值，避免级联故障
- 降级策略：返回兜底数据 / 缓存数据，保障核心功能可用

**限流：**
- 接口级别令牌桶限流，防止流量突增打垮数据库
- 热点用户识别 + 单独限流规则

**幂等设计：**
- 支付接口防重：Token 机制 + 数据库唯一键双重保障
- MQ 消费幂等：消息 ID 去重表

### 2.3 数据一致性类

**分布式事务：**
- 订单 + 库存 + 积分跨服务，使用 Seata AT 模式保障最终一致
- 本地消息表 + 定时补偿，保障核心扣款与通知的最终一致

**缓存与数据库一致性：**
- Cache Aside 模式：先删缓存，更新 DB，再延迟双删
- Binlog 订阅（Canal）+ MQ 异步更新缓存

### 2.4 系统设计类

**秒杀系统：**
- Redis 预减库存 + 消息队列异步下单，支撑 5w QPS 秒杀场景
- 库存数据预热到 Redis，避免请求穿透到 DB

**搜索系统：**
- MySQL → Elasticsearch 迁移，支持全文检索、商品搜索建议
- Canal 同步 MySQL 数据到 ES，数据延迟 < 1s

## 三、难点提炼方向

### 3.1 分布式场景下的数据一致性

**难点描述（示例）：**
> 订单支付成功后，需要同时更新订单状态、扣减库存、增加积分、发短信通知。这四步跨三个服务、两个数据库，如何保证不出现"扣了钱但库存没减"的情况？

**解决思路：**
- RocketMQ 事务消息：支付成功作为半消息，本地事务成功后 commit
- Seata AT 模式：`@GlobalTransactional` 注解，底层自动生成 Undo Log 回滚
- Saga 补偿：各步骤失败后执行对应补偿操作

### 3.2 缓存热点问题

**难点描述（示例）：**
> 大促时某款爆款商品 QPS 达到 10w+，Redis 单个 key 的访问量打满了单分片带宽，出现热点 key 问题。

**解决思路：**
- 本地缓存（Caffeine）拦截热点读，大幅减少 Redis 访问
- Redis key 分片：`hot_product:123:shard_{0~9}` 分散到多个 slot
- 读时随机选一个分片，写时广播所有分片

### 3.3 超卖问题

**难点描述（示例）：**
> 秒杀场景下，库存只有 100 个，但同时有 1000 个请求并发扣减，如何防止超卖？

**解决思路：**
1. **数据库乐观锁**：`UPDATE stock SET count=count-1 WHERE id=? AND count>0`
2. **Redis + Lua 原子扣减**：保证读-改-写的原子性
3. **Redis 预减 + DB 异步落库**：秒杀流量在 Redis 层拦截，只有预减成功的请求才进入 MQ 异步下单

### 3.4 分库分表的迁移难点

**难点描述（示例）：**
> 订单表从单表 5 亿数据，迁移到 16 库 × 16 表 = 256 张分表，如何做到不停机迁移？

**双写方案：**
```
阶段1：双写（同时写旧表和新分表）+ 读旧表
阶段2：数据迁移（历史数据批量迁移）+ 校验
阶段3：双写 + 读新分表（灰度切流）
阶段4：下线旧表写入，仅写新分表
```

## 五、常见反问与应对

**"如果让你重新设计，你会怎么改？"**
> 诚实指出当前方案的局限，说明如果重来会选择什么更好的方案，体现技术成长和反思能力。

**"你在这个项目里具体做了什么？"**
> 明确自己的职责边界，避免夸大。"我负责了 XXX 模块的设计和实现，参与了 XXX 的讨论"比"我设计了整个系统"更可信。

**"遇到最难的问题是什么？怎么解决的？"**
> 选一个真实遇到过的问题，描述排查过程（看日志/监控/代码 Review），展示解决问题的方法论，比结果更重要。
